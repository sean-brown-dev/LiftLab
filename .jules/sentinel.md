## 2024-05-17 - Prevent Sensitive Data Logging in AI Clients
**Vulnerability:** The AI client implementation (`ProgramGenerationAiClient.kt`) was logging the complete prompt sent to the LLM as well as the complete raw JSON response using `Log.d`.
**Learning:** Even if data is not considered a high-level secret (like an API key), logging complete conversational turns, prompts, and large payloads from AI generation processes can unintentionally leak user-specific data, patterns, or PII into the system logcat. This data could then be exposed if logs are aggregated or accessed by other means.
**Prevention:** Avoid logging complete request/response payloads in production code, especially those interacting with third-party APIs or AI models where user context is included. Only log necessary metadata or error details needed for debugging without exposing the core data payload.

## 2024-05-18 - Prevent Sensitive Data Leakage in Crashlytics Logging
**Vulnerability:** The application was inadvertently logging entire domain objects (specifically lists of active `Program` objects) into `FirebaseCrashlytics` and `Logcat` when encountering edge cases during active program checks and remote syncs.
**Learning:** Passing complete domain objects to logging tools implicitly serializes them via `.toString()`. If these objects contain user-defined data (like program names or custom configurations), this constitutes a PII/sensitive data leak into the error tracking system. This is a common pattern when quickly writing exception messages like `Exception("Multiple active programs: $activePrograms")`.
**Prevention:** When logging exceptions or application state, strictly map domain models to their primary identifiers (e.g., `.map { it.id }`) or aggregate counts. Never embed full domain objects or unvetted collections directly into log strings or exception messages passed to third-party services.
